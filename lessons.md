# Lessons

- 用户明确纠偏后，先回到需求本身，不要为了压耗时私自加“少字走特殊接口”这类分流规则。
- 同步翻译变慢先查上游接口真实耗时、重试、返回格式和缓存命中，不要先改业务路径掩盖问题。
- 不要把 OpenClaw 的 `translation_agent`、底层 `model` 和返回里的 `provider_model` 混成一件事；当前仓库里 OpenClaw 传的是 `chemical-news-translator`。
- 改 OpenClaw 默认 agent 这种事，先想清楚是不是要改，别只凭猜测替换默认值。
- 同仓库的本地到远端代码同步必须走 git；如果按规范需要开分支才能推送，就先问用户，不准为了省事直接 `scp` 覆盖远端文件。
- 涉及 GitHub 发布链路时，先分清是在本机还是远端环境执行；不要只查本机 `gh` 就下结论说没装。
- 已经加入 `.gitignore` 的文件如果此前被 Git 跟踪过，必须执行 `git rm --cached` 解除跟踪；只改忽略规则不会生效。
- 品牌图标类需求，用户已经给了官网或参考图，就按源图还原；不要自己脑补成别的符号或品牌元素。
- 用户给的是具体头像图时，不要画“差不多”的风格版，直接按那张图的轮廓和留白去还原。
- GitHub CLI 优先看远端环境；本项目里 `gh` 在远端，不要只在本机查完就下结论。
- 用户说“OpenClaw 继续调这个 agent，只改 agent 里面的模型”时，先去远端 `~/.hermes/profiles/chemical-news-translator` 查真实运行模型，不要继续在 `caphub` 仓库里改 OpenClaw 请求参数。
- 用户指定某个任务 UUID 做验证时，必须优先复用该任务的原始内容或从该任务派生同内容重跑，不要擅自换成短样例文本。
- 任务追踪页这类状态页，用户更关心“原文/译文并排对照 + 当前 agent + 下一步动作”，不要只堆状态卡片和时间线。
- 任务成功页不要再额外塞“查看结果”这种重复按钮；进度区也别拆四个状态卡片，直接用当前阶段顶替整体进度文案更清楚。
- HTML 分段翻译里只查残留中文不够，`__HTML_ENTITY_n__` 这种实体占位符残留也必须直接判成脏结果并触发重试。
