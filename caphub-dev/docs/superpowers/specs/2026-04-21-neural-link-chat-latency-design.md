# Neural Link Chat 延迟收缩设计

## 目标

- 把首页聊天从“能回但很慢”收缩到“能正常连续追问”。
- 直接暴露上游空响应和超时，不再用含糊提示掩盖。
- 不改远端 Hermes profile，不碰部署结构，只改当前仓库前后端。

## 方案

### 后端

- 把聊天 system prompt 从“整份项目文件拼接”改成“精简项目快照”。
- 模型列表查询加缓存，避免每轮聊天都额外打两次 `/v1/models`。
- 服务端只保留最近 6 条历史消息，并截短单条内容，防止多轮对话越聊越重。
- 上游返回空文本时直接按错误处理，返回明确提示。

### 前端

- 前端同步只上传最近 6 条历史消息，单条再截短，减少无效 payload。
- 捕获服务端错误时优先显示服务端返回文案，不再统一写成“连接中断”。

## 验证

- 后端 Feature Test 覆盖精简 prompt 和空响应报错。
- 前端 Vitest 覆盖聊天失败时展示服务端错误文案。
- 实际页面再跑两轮聊天，确认能连续返回。
